# План добавления функций кнопок

На основании [PROJECT.md](./PROJECT.md).

## Текущее состояние

| Компонент | Статус |
|-----------|--------|
| Поле ввода URL | Готово |
| Кнопки «О чем статья?», «Тезисы», «Пост для Telegram» | Есть в UI, AI не подключён |
| Парсинг HTML (`lib/parseArticle.ts`, `/api/parse`) | Готово |
| OpenRouter + Deepseek (`lib/openrouter.ts`) | Готово (используется для «Перевод») |
| Блок «Результат» | Готово |

Сейчас три кнопки вызывают только парсинг и показывают JSON `{ date, title, content }`. Нужно подключить AI и выводить осмысленный ответ для каждого действия.

---

## Порядок действий

### Шаг 1. Вынести общую логику AI-обработки

1. Создать `lib/aiActions.ts` с типом действия: `summary` | `theses` | `telegram`.
2. Для каждого действия описать **system prompt** и **user prompt** на русском:
   - **summary** — кратко: о чём статья, основная мысль, вывод (3–5 предложений).
   - **theses** — структурированный список тезисов (маркированный список, 5–10 пунктов).
   - **telegram** — готовый пост: заголовок, краткая суть, 1–2 абзаца, ссылка на источник.
3. Общая функция `runAiAction(action, article, sourceUrl)`:
   - принимает распарсенную статью (`title`, `content`, `date`);
   - обрезает `content` до лимита токенов (~12 000 символов);
   - вызывает `chatCompletion` из `lib/openrouter.ts`;
   - возвращает текст ответа.

### Шаг 2. Создать единый API-маршрут

1. Добавить `app/api/analyze/route.ts`.
2. Принимать POST `{ url, action }`, где `action` — `summary` | `theses` | `telegram`.
3. Внутри маршрута:
   - валидировать URL;
   - вызвать `parseArticleFromUrl(url)`;
   - проверить, что `content` не пустой;
   - вызвать `runAiAction(action, article, url)`;
   - вернуть `{ action, result }`.
4. Обработать ошибки: нет ключа API, пустая статья, таймаут OpenRouter.

### Шаг 3. Подключить кнопки в интерфейсе

1. В `app/components/ArticleAnalyzer.tsx` изменить `handleAction`:
   - для `summary`, `theses`, `telegram` — запрос на `/api/analyze`;
   - передавать `{ url, action }`.
2. Обновить `LOADING_LABELS`:
   - «О чем статья?» → «Анализ статьи...»
   - «Тезисы» → «Формирование тезисов...»
   - «Пост для Telegram» → «Подготовка поста...»
3. Выводить `result` как обычный текст (не JSON) в блоке «Результат».
4. Бейдж над результатом оставить — показывает выбранное действие.

### Шаг 4. Настроить промпты под качество ответа

1. **«О чем статья?»** — ответ на русском, без пересказа каждого абзаца.
2. **«Тезисы»** — нумерованный или маркированный список, каждый пункт — одна мысль.
3. **«Пост для Telegram»** — короткий пост до ~1 500 символов, эмодзи по желанию, в конце URL источника.

Промпты хранить в `lib/aiActions.ts`, чтобы править без изменения API и UI.

### Шаг 5. Рефакторинг (по желанию)

1. Объединить `/api/translate` и `/api/analyze` в один `/api/process` с полем `action: summary | theses | telegram | translate`.
2. Вынести валидацию URL в `lib/validateUrl.ts` — сейчас она дублируется в маршрутах.
3. Кэшировать результат парсинга на время сессии на клиенте, чтобы при переключении кнопок не парсить статью заново.

### Шаг 6. Проверка

1. Запустить `pnpm.cmd dev`.
2. Протестировать на 2–3 реальных англоязычных статьях.
3. Проверить все три кнопки:
   - ответ на русском;
   - формат соответствует действию;
   - ошибки (пустой URL, недоступный сайт, нет API-ключа) показываются в UI.
4. Убедиться, что `.env.local` с `OPENAI_API_KEY` и `OPENAI_BASE_URL` подхватывается после перезапуска dev-сервера.

---

## Итоговая схема

```
URL → [кнопка] → /api/analyze → parseArticle → OpenRouter (Deepseek) → результат в UI
```

| Кнопка | API action | Ожидаемый результат |
|--------|------------|---------------------|
| О чем статья? | `summary` | Краткое описание статьи |
| Тезисы | `theses` | Список ключевых тезисов |
| Пост для Telegram | `telegram` | Готовый текст поста |
